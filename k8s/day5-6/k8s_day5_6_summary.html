2. Architecture

The app follows a 3-tier pattern: React frontend → Flask backend → PostgreSQL database, all running in Kubernetes on EKS, exposed via an AWS Application Load Balancer.

3. What Was Done & Why

Terraform Split into Two Modules

Infrastructure is split into two separate Terraform roots:

  • core-cluster/ — VPC, subnets, EKS cluster, node groups. Changes here are rare and costly.
  • k8s-services/ — Helm releases (ArgoCD, monitoring, LB controller), Route53 records, ACM certs. Changes here are frequent.

Why split: Running a single terraform apply on everything would recreate the cluster on minor service changes. The split means cluster rebuilds are opt-in.

3 Worker Nodes Instead of 1 or 2

t3.small instances support a maximum of 11 pods per node (AWS ENI limit). The full stack (system pods + monitoring + ArgoCD + app pods) needed more than 11 pods total, so 3 nodes were used to spread the load.

DB Migration as a Kubernetes Job

Initially the migration script was part of the container's startup command (CMD). This caused problems: if a pod restarted, migrations ran again and sometimes failed. Moving it to a dedicated Kubernetes Job means it runs once, completes, and does not run again on pod restarts.

Prometheus & Grafana Resource Reduction

The default Helm chart values for Prometheus requested 2GB of memory — far too much for t3.small nodes (which only have 2GB total). Resource requests were reduced: Prometheus to 512Mi, Grafana to 256Mi, and the Helm install timeout was extended to 900 seconds to allow pods time to schedule.

Terraform State in S3

Remote state stored in S3 bucket state-bucket-768093818017 with two keys:

  • eks-may-2026/eks-core-cluster/terraform.tfstate
  • eks-may-2026/k8s-services/terraform.tfstate

This allows team members (and future sessions) to share the same state without conflicts.

ACM Certificates

Two separate ACM certificates were created — one for the main app domain and one for ArgoCD — because they are managed by different Terraform modules and ALBs.

4. Errors Encountered & Fixes

Every error below was hit during the actual assignment. These are real fixes, not theoretical ones.
Error 1 — Windows CRLF Breaking Shell Script in Container

The migrate.sh database migration script was written on Windows, which saved it with \r\n (CRLF) line endings. When the Linux container tried to execute it, bash read the \r as part of the command and threw:

bash: $'\r': command not found
Fix

Added a sed stripping step in the Dockerfile, before the script is executed:

RUN sed -i 's/\r//' migrate.sh

This strips all carriage return characters from the script at image build time, making it safe to run on Linux regardless of how the file was edited on Windows.

Error 2 — t3.small Pod Limit Exceeded (Pending Pods)

Pods kept staying in Pending state with the event:

0/2 nodes are available: 2 Too many pods.

AWS limits t3.small to 11 pods per node due to ENI/IP constraints. With 2 nodes, system pods alone consumed most of the capacity.

Fix

Increased the node group from 2 to 3 nodes in the Terraform EKS config. 3 nodes gave 33 pod slots, enough to run all system, monitoring, ArgoCD, and app pods comfortably.

Error 3 — Prometheus Helm Install Timeout / OOMKilled

The default Prometheus Helm chart requested 2Gi memory, which exceeded what a t3.small node could provide. Install would timeout, and pods would be OOMKilled or stay in Pending.

Fix

Overrode resource values in the Helm values file:

prometheus:
  server:
    resources:
      requests:
        memory: "512Mi"
grafana:
  resources:
    requests:
      memory: "256Mi"

Also increased the Helm install timeout to 900 seconds (--timeout 900s) to give pods time to pull images and start on constrained nodes.

Error 4 — Helm "Cannot Re-use a Name That Is Still in Use"

After a failed Helm install, retrying gave:

Error: cannot re-use a name that is still in use

Helm stores release state as a Kubernetes Secret. A failed install leaves behind a secret in a failed state, blocking reinstall.

Fix

Delete the stuck Helm release secret manually, then re-run the install:

kubectl delete secret sh.helm.release.v1.<release-name>.v1 -n <namespace>
Error 5 — Terraform State Lock Not Released

After a cancelled or failed terraform apply, subsequent runs failed with a state lock error, preventing any further Terraform operations.

Fix

Deleted the lock file directly from S3:

aws s3 rm s3://state-bucket-768093818017/eks-may-2026/k8s-services/terraform.tfstate.tflock
Error 6 — ECR Docker Login Failing on Windows PowerShell

The standard Linux pipe approach for ECR login:

aws ecr get-login-password | docker login --username AWS --password-stdin <url>

Did not work reliably in Windows PowerShell due to how PowerShell handles piped stdin to native executables.

Fix

Used a variable instead of a pipe:

$token = aws ecr get-login-password --region ap-south-1
docker login --username AWS --password $token 768093818017.dkr.ecr.ap-south-1.amazonaws.com
Error 7 — ALB Route53 Alias Needs Hardcoded Hosted Zone ID

When creating a Route53 alias record pointing to the ALB for ArgoCD, Terraform's aws_lb data source does not always expose the ALB's hosted zone ID, or the value is not available at plan time.

Fix

Hardcoded the hosted zone ID for ap-south-1 ALBs in argocd.tf:

zone_id = "ZP97RAFLXTNZK"  # ap-south-1 ALB hosted zone ID

This is a static AWS value that does not change for a given region.

Error 8 — DB Migration Running on Every Pod Restart

Migration was originally placed in the container's CMD. Every time a pod restarted (due to a crash, rolling update, or scaling), the migration ran again — causing errors when it tried to re-create already-existing tables.

Fix

Moved the migration into a dedicated Kubernetes Job (migration-job.yaml). A Job runs to completion once and does not re-run on pod restarts. The application deployment was updated to remove the migration from its startup command.

5. Deployment Order

Order matters — each step depends on the previous one being ready.

  1. terraform apply in k8s/EKS/core-cluster/ — creates VPC + EKS cluster
  2. terraform apply in k8s/EKS/k8s-services/ — installs ArgoCD, monitoring, LB controller, Route53 records
  3. Create RDS instance manually: identifier devopsdozo-db, Postgres 15.13, db.t3.micro
  4. kubectl apply manifests in this order:
    1. namespace.yaml
    2. configmap.yaml
    3. secrets.yaml
    4. backend.yaml
    5. frontend.yaml
    6. ingress-https.yaml
    7. migration-job.yaml

6. Destroy Order

Reverse of creation — must be done in this order to avoid orphaned AWS resources that block deletion.

  1. kubectl delete namespace 3-tier-app-eks — removes ALB and app pods first
  2. Delete Route53 A record for devopsdozo.adishrivtech.in manually in AWS Console
  3. Delete ACM certificate for devopsdozo.adishrivtech.in manually
  4. Delete RDS instance devopsdozo-db manually
  5. terraform destroy in k8s/EKS/k8s-services/
  6. terraform destroy in k8s/EKS/core-cluster/
